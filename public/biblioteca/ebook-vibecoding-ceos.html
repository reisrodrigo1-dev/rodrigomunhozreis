01
A real

Por que um CEO precisa entender disso

Você não precisa virar programador. Mas, se a sua empresa vai construir software com IA, você precisa saber o que separa um produto sólido de uma bomba-relógio. Senão, você paga a conta sem nem entender por quê.

Vibecoding — construir software conversando com a IA — é a maior alavanca de produtividade que já apareceu. Dá pra tirar uma ideia do papel em dias. Mas a velocidade tem um lado perigoso quando ninguém revisa o que a máquina produz.

⚠ Os números que assustam
  • Código gerado por IA tem 2,74x mais vulnerabilidades que código humano (Veracode, 2025).
  • 61% do código de IA funciona, mas só 10,5% passa numa revisão de segurança.
  • Em 5.600 apps "vibecodados": 400+ segredos expostos e 175 vazamentos de dados pessoais.

Traduzindo para a linguagem do dono: vazamento de dado de cliente é multa de LGPD; sistema inseguro é processo e reputação; arquitetura ruim é retrabalho caro. Tudo isso é evitável — com método. É o que este guia te dá.

02
Mentalidade

Vibecoding com engenharia

O inimigo nunca foi a IA. É o vibecoding às cegas: aceitar tudo o que a máquina cospe sem entender.

A virada de chave é simples: trate a IA como um estagiário brilhante e rápido — não como um oráculo infalível. Estagiário entrega rápido, mas tudo passa por revisão. Quatro princípios guiam isso:

1. Plano antes do código

Você (ou a IA, sob sua direção) decide arquitetura e escopo antes de gerar qualquer coisa.

2. Pedido como especificação

Pedido vago gera lixo. Pedido preciso gera código revisável.

3. Revisão sempre

Nada é aceito sem ler. Use um agente especialista para revisar (cap. 8).

4. Você manda na qualidade

Nada vai para produção sem passar pelos cuidados de segurança.

"Velocidade sem engenharia não é economia. É dívida com juros — e a conta chega na pior hora."

O ritmo certo

O fluxo de trabalho seguro

Todo projeto bem-feito segue o mesmo ciclo. Decore esta ordem — ela evita 90% dos desastres:

Planejar

Antes de codar, peça à IA um plano em fases (MVP primeiro), a arquitetura e os riscos. Não comece pelo código.

Construir em pedaços pequenos

Uma funcionalidade por vez. Pedaços pequenos são fáceis de revisar e de desfazer.

Revisar (Protocolo de 5 Camadas)

Entender, Ler, Blindar (segurança/LGPD), Testar e Versionar — antes de aceitar (cap. 8 e 9).

Versionar (commit)

Commits pequenos e frequentes. Sempre git pull antes de enviar (cap. 5).

Publicar (deploy)

Subir para a Vercel só depois de passar pelos cuidados. Configure as variáveis no ambiente certo (cap. 7).

💡 Regra de ouro

"Funcionou na tela" não é "está pronto". Pronto é depois de revisado, testado e com os dados protegidos.

05
Versionamento

Git & GitHub sem dor

Git guarda o histórico do seu código; o GitHub é onde ele fica salvo e compartilhado. Dois cuidados aqui evitam vazamentos e dores de cabeça enormes.

1. Sempre dê git pull antes de subir

Antes de enviar suas mudanças (push), baixe as últimas alterações com git pull. Isso evita sobrescrever o trabalho de outra pessoa (ou de você mesmo em outra máquina) e conflitos feios.

git pull # baixa o que mudou ANTES de subir git add . git commit -m "minha alteração" git push # agora sim, envia

2. NUNCA suba .env nem arquivos sensíveis

O arquivo .env guarda suas senhas e chaves secretas. Se ele for parar no GitHub, qualquer um pode roubar seus acessos. O mesmo vale para chaves e service accounts. Garanta isso no .gitignore:

# .gitignore — o que NUNCA versionar node_modules .next .env .env* *.pem *service-account*.json # .md — só se você não quiser versionar documentos *.md
⚠ Peça isso à IA, sempre

Dê à IA a regra permanente: "nunca inclua arquivos .env, chaves, segredos ou .md no que for subir, e sempre confirme o que vai ser enviado antes do push." Está pronto na biblioteca (prompt 14).

Se um segredo já foi para o git, troque a chave (revogue a antiga) — apenas apagar o arquivo não basta, pois ele fica no histórico.

Backend

Firebase com segurança

O Firebase (do Google) te dá um backend pronto: login (Authentication), banco de dados (Firestore) e arquivos (Storage). Três cuidados são inegociáveis.

Config em variáveis de ambiente

A configuração do Firebase vai no .env.local (nunca colada no código). No padrão Next.js, com o prefixo NEXT_PUBLIC_.

Regras de segurança

A apiKey do Firebase web é pública por design — a segurança real são as Regras do Firestore e do Storage: definem quem pode ler e escrever cada coisa. Sem regras, seu banco fica aberto.

App Check + LGPD

Ative o App Check (bloqueia acesso de fora do seu app) e mapeie quais dados pessoais você guarda, onde e quem acessa.

Exemplo de regra (Firestore)
match /leads/{id} { allow create: if true; // qualquer um cria (captura) allow read: if isAdmin(); // só admin lê }

Use o prompt 5 da biblioteca: ele já pede a conexão completa + as regras + o App Check, e você só cola a sua config.

07
Publicação

Deploy na Vercel

A Vercel publica seu site no ar em minutos, conectada ao seu GitHub. Cada vez que você dá push, ela atualiza o site sozinha.

Suba o código no GitHub

Com os cuidados do capítulo 5 (sem .env, com git pull antes).

Importe o repositório na Vercel

vercel.com → Add New Project → escolha o repositório. Ela detecta o Next.js sozinha.

Configure as variáveis de ambiente

Em Settings → Environment Variables, recadastre as MESMAS chaves do seu .env.local. (O .env não vai pelo git — por isso você precisa colá-las aqui.)

Deploy e domínio

Clique em Deploy. Depois, em Settings → Domains, aponte o seu domínio próprio.

⚠ O erro nº1 no deploy

"Funciona no meu PC mas quebra na Vercel" quase sempre é variável de ambiente faltando. Como o .env não sobe para o git, você precisa cadastrá-las também na Vercel.

Como falar com a IA

P.E.D.E.R. + agentes especialistas

Pedido vago = resposta ruim. A receita P.E.D.E.R. garante bons pedidos:

Papel Aja como um arquiteto de software sênior. Exemplo/Contexto Quero construir [produto] para [público]. Desejo Antes de codar, monte o plano em fases. Estilo Em tópicos, com riscos destacados. Refinar Me faça perguntas se faltar algo.

Agentes especialistas: a virada de chave

Em vez de uma IA genérica, peça para ela assumir um papel especialista e salvar esse papel para reusar. Os mais valiosos para um dono de empresa:

  • Agente Arquiteto — monta o plano e a arquitetura antes de qualquer código.
  • Agente Revisor de Segurança — lê o código e caça segredos expostos, falhas de autenticação e dados desprotegidos.
  • Agente de Deploy — guia GitHub e Vercel com os cuidados certos.

Os prompts 1, 3 e 9 da biblioteca criam exatamente esses agentes. Use-os como ponto de partida de todo projeto.

Antes de respirar aliviado

Checklist de segurança

Rode esta lista antes de subir qualquer coisa para produção. Se algum item falhar, pare.

✅ Segredos.env e chaves fora do git (confirmado no .gitignore).
✅ Gitgit pull feito antes do push; nenhum .env/.md sensível subindo.
✅ FirebaseRegras do Firestore e do Storage publicadas e restritivas. App Check ativo.
✅ Dados (LGPD)Você sabe quais dados pessoais coleta, onde ficam e quem acessa.
✅ RevisãoCódigo passou pelo Protocolo de 5 Camadas / agente revisor.
✅ VercelVariáveis de ambiente recadastradas; build sem erros.
✅ BackupVocê consegue voltar atrás (histórico do git, export do banco).
A pergunta de ouro do dono

Antes de aceitar qualquer mudança importante, pergunte à IA: "o que isso faz, o que pode quebrar, e isso toca em segurança, dados ou pagamentos?" (prompt 10).

Prompts prontos, completos, para colar na IA em cada etapa — do plano ao deploy. Troque o que está [entre colchetes] pelo seu caso. Na versão digital, clique no prompt para copiar.

1

Plano & arquitetura

01
Montar o plano do projeto (agente arquiteto)
Aja como um arquiteto de software sênior. Quero construir [descreva o produto/ideia] para [público-alvo]. Antes de escrever QUALQUER código: (1) me faça as perguntas necessárias para entender o escopo; (2) proponha um plano em fases, começando por um MVP enxuto; (3) sugira a arquitetura, as telas e os módulos; (4) liste os principais riscos e cuidados de segurança. Não comece a codar ainda — só o plano. Salve esse papel de "arquiteto" para eu reusar neste projeto.
02
Sugerir a stack (tecnologias)
Aja como CTO. Para o projeto [descreva], sugira a stack mais simples e segura para um time pequeno, equilibrando custo e velocidade. Inclua: framework de frontend, estilização, backend/banco, autenticação, hospedagem e (se preciso) pagamentos. Para cada escolha, explique o porquê em 1 linha e dê 1 alternativa. Se eu não tiver preferência, use o padrão: Next.js + Tailwind + Firebase + Vercel.
2

Construção & revisão

03
Criar agente especialista para analisar o código
Crie e salve um agente/persona especialista em revisão de código e segurança para este projeto. Ele deve, em todo código que eu mostrar: (1) explicar em português simples o que o código faz; (2) apontar vulnerabilidades — segredos expostos, falta de validação de entrada, problemas de autenticação/autorização, exposição de dados; (3) apontar problemas de arquitetura e dívida técnica; (4) sugerir as correções, em ordem de gravidade — sem reescrever tudo sem me avisar. A partir de agora, use essa persona quando eu pedir "revisar".
04
Scaffolding inicial com segurança
Aja como engenheiro sênior. Crie a base do projeto usando [stack — ex.: Next.js + Tailwind]. Já configure: estrutura de pastas limpa, variáveis de ambiente em um arquivo .env.local (NUNCA valores hardcoded no código), e um .gitignore que ignore node_modules, .next, todos os .env* e arquivos sensíveis (*.pem, service-account*.json). Explique cada arquivo de configuração em 1 linha. Não suba nada ainda.
05
Antes de aceitar qualquer alteração
Antes de eu aceitar essa mudança: explique em português simples o que ela faz, o que pode quebrar, e se ela toca em segurança, dados pessoais ou pagamentos. Se tocar, detalhe o cuidado necessário. Não altere mais nada além do que pedi.
3

Firebase

06
Conectar o Firebase com segurança
Aja como especialista em Firebase. Conecte este projeto ao Firebase usando Authentication, Firestore e Storage. Requisitos obrigatórios: (1) configuração via variáveis de ambiente (NEXT_PUBLIC_*), NUNCA no código; (2) inicialização que não recria o app no hot-reload; (3) me entregue as REGRAS de segurança do Firestore e do Storage, restritivas (leitura/escrita só para quem pode); (4) explique, em passos, como ativar o App Check no console. Aqui está minha config do Firebase: [cole a config do console].
07
Login só para mim (admin)
Quero uma área de administração que só EU acesse. Implemente login com e-mail/senha (Firebase Auth), proteja as rotas de admin no app e reforce no servidor com as regras do Firestore (só o e-mail [seu e-mail] pode escrever). Me explique como criar o meu usuário admin no console.
4

Git, GitHub & Vercel

08
Configurar o .gitignore (não subir .env/.md)
Revise e ajuste meu .gitignore para NUNCA versionar: node_modules, .next/out/build, TODOS os .env*, arquivos .md (por opção minha), e segredos (*.pem, *service-account*.json). Depois, verifique se algum desses já está sendo rastreado pelo git e me diga como removê-lo do versionamento SEM apagar do meu PC (git rm --cached). Confirme a lista final do que será e do que NÃO será enviado.
09
Subir no GitHub (com git pull antes)
Me guie, passo a passo e com os comandos exatos, para subir este projeto no GitHub: (1) git init; (2) criar o repositório remoto; (3) adicionar o remote origin; (4) ANTES de cada envio, sempre rodar "git pull" para não sobrescrever nada; (5) git add, commit e push. IMPORTANTE: antes do push, confirme comigo que nenhum arquivo .env, segredo ou .md sensível será enviado.
10
Deploy na Vercel
Me guie a publicar este projeto na Vercel, passo a passo e com cliques: (1) conectar o repositório do GitHub; (2) onde e como cadastrar as variáveis de ambiente (as mesmas do meu .env.local, que não foi para o git); (3) como configurar meu domínio próprio; (4) onde ver os logs caso o deploy dê erro. Aponte o erro mais comum (variável de ambiente faltando) e como evitá-lo.
5

Segurança & produção

11
Revisão de segurança (Protocolo de 5 Camadas)
Aja como auditor de segurança. Revise este código aplicando 5 camadas: (1) Entender o que ele faz; (2) Ler e descrever cada parte; (3) Blindar — procure segredos expostos, falta de validação de entrada, falhas de autenticação/autorização e exposição de dados pessoais (LGPD); (4) Testar — sugira os testes essenciais; (5) Versionar — confirme que nada sensível vai para o git. Liste os problemas por gravidade (crítico/alto/médio) e a correção de cada. Código: [cole o código].
12
Checklist de pré-deploy
Monte um checklist de pré-deploy para este projeto e me diga o que ainda falta: variáveis de ambiente configuradas (local e na Vercel), .env e segredos fora do git, regras do Firestore/Storage publicadas, App Check ativo, testes passando, build sem erros e um plano de backup/rollback.
13
LGPD e dados pessoais
Aja como consultor de LGPD (isto não substitui um advogado). Analise este projeto e liste: quais dados pessoais ele coleta, onde ficam armazenados, quem tem acesso, e o que eu preciso para estar em conformidade (consentimento, política de privacidade, segurança dos dados). Aponte os riscos e como mitigá-los. Contexto do projeto: [descreva].
14
Regra permanente (não subir arquivos sensíveis)
Regra permanente para este projeto: ao me ajudar a versionar ou subir código, NUNCA inclua arquivos .env, chaves, service accounts ou .md sensíveis. Sempre me mostre e confirme a lista do que será enviado ANTES de qualquer push. Se eu esquecer, me lembre.
15
Debugar um erro com segurança
Estou com este erro: [cole o erro]. Aja como engenheiro sênior: explique a causa provável em português simples, proponha a correção mais segura (sem gambiarra), diga como evitar que volte e confirme que a correção não introduz risco de segurança. Mude só o necessário.

Para levar com você

O dono que entende, decide melhor

Você não precisa escrever código. Mas, sabendo o que é um bom processo, você para de ser refém — de fornecedor, de IA ou de um time sem método.

  • Plano antes do código. Sempre.
  • Revisão sempre — com um agente especialista de segurança.
  • git pull antes de subir, e nunca versionar .env, segredos ou .md sensíveis.
  • Firebase com regras, Vercel com variáveis, e os dados em conformidade com a LGPD.

"Vibecoding não é atalho para quem não sabe o que está fazendo. É alavanca para quem tem método."
